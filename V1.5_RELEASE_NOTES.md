# Agile V™ Skills Library v1.5: Product Development Extensions

**Release Date:** February 26, 2026  
**Version:** 1.5  
**License:** CC-BY-SA-4.0

## Summary

Version 1.5 introduces **six new skills** that extend Agile V's rigor across the full product development lifecycle — from discovery through production operations. These skills address critical gaps while maintaining Agile V's core philosophy: **everything remains traceable, verifiable, and gateable**.

All new skills are **context-optimized** following v1.3 patterns (53% size reduction), ensuring minimal context window consumption while preserving complete functionality.

## What's New

### 1. Discovery-Analyst Skill (`discovery-analyst/`)

**Purpose:** Bridges messy discovery inputs (user interviews, feedback, tickets, research) to formal requirements.

**Key Features:**
- Converts observations → insights → hypotheses → candidate requirements
- Tracks assumptions with validation plans
- Plans experiments (A/B tests, user studies, spikes) with success criteria
- Introduces **Human Gate 0** (Discovery Approval) before requirements formalization
- Full traceability: Every REQ-XXXX must cite discovery artifact or be marked "stakeholder directive"

**New Artifacts:**
- `.agile-v/DISCOVERY_LOG.md` (OBS-XXXX, INS-XXXX, HYP-XXXX, ASM-XXXX)
- `.agile-v/EXPERIMENTS.md` (EXP-XXXX, validation results)

**Why It Matters:** Reduces Logic Gatekeeper's burden of turning subjective goals into metrics later. Maintains "no silent assumptions" stance from the start.

**Example Traceability Chain:**
```
OBS-0001 (User abandons checkout) → INS-0001 (Multi-step friction) → 
HYP-0001 (Guest checkout reduces abandonment) → EXP-0001 (A/B test) → 
CANDIDATE-0001 → REQ-0001 (Guest checkout feature)
```

### 2. Threat-Modeler Skill (`threat-modeler/`)

**Purpose:** STRIDE-based threat modeling and privacy impact assessment to generate security/privacy requirements.

**Key Features:**
- STRIDE analysis (Spoofing, Tampering, Repudiation, Info Disclosure, DoS, Elevation of Privilege)
- Privacy Impact Assessment (PII inventory, GDPR/CCPA compliance, cross-border transfers)
- Generates CRITICAL priority security requirements *before* code exists (shift security left)
- Integrates with Red Team Verifier for security testing (SQLMap, OWASP ZAP, penetration tests)

**New Artifacts:**
- `.agile-v/THREAT_MODEL.md` (THREAT-XXXX, mitigations, severity matrix)
- `.agile-v/PRIVACY_IMPACT_ASSESSMENT.md` (PII-XXXX, data flows, GDPR compliance)

**Why It Matters:** Shifts security from "compliance artifact" to "requirements constraint." Security failures become CRITICAL defects that block release, not post-production surprises.

**Example:**
```
THREAT-0001 (SQL Injection) → CANDIDATE-SEC-0001 (Parameterized queries) → 
REQ-0042 (CRITICAL) → ART-0042 (ORM implementation) → 
TC-0142 (SQLMap scan) → Red Team executes → Pass → Gate 2
```

### 3. UX-Spec-Author Skill (`ux-spec-author/`)

**Purpose:** Converts UX/design intent into testable design specifications that become requirements.

**Key Features:**
- User flows with edge cases (loading, error, empty, offline states)
- Accessibility requirements (WCAG 2.1 AA, keyboard navigation, screen readers) — all CRITICAL priority
- Performance budgets (load time, interaction latency, frame rate) with measurement methods
- Responsive design constraints (breakpoints, touch targets, mobile-first)
- Design decisions = requirements, not suggestions

**New Artifacts:**
- `.agile-v/DESIGN.md` (User flows, interaction patterns, A11Y requirements, performance budgets)
- A11Y-XXXX requirements (WCAG compliance, always CRITICAL)
- PERF-XXXX requirements (performance targets with verification methods)

**Why It Matters:** Elevates design from "nice-to-have prose" to testable constraints. Accessibility and performance become measurable criteria the Red Team can verify.

**Example:**
```
UF-0001 (Login flow) → IP-0001 (Button specs: 44×44pt, contrast 4.5:1) → 
A11Y-0001 (Keyboard accessible, WCAG 2.1.1) → CANDIDATE-0023 → 
REQ-0023 (CRITICAL) → TC-0123 (axe DevTools, manual keyboard test) → Verified
```

### 4. Agile-V Product Owner Skill (`agile-v-product-owner/`)

**Purpose:** REQ-aware Product Owner for sprint-based delivery with full traceability.

**Key Features:**
- Every backlog item (BL-XXXX) maps to one or more REQ-XXXX
- Sprint planning with capacity tracking and velocity forecasting
- INVEST story validation (every story must be Independent, Negotiable, Valuable, Estimable, Small, Testable)
- Retrospectives generate CR-XXXX when requirements need adjustment
- Sprint artifacts feed into Agile V lifecycle (cycle scope, risk updates, approval records)

**New Artifacts:**
- `.agile-v/BACKLOG.md` (BL-XXXX with REQ mappings)
- `.agile-v/sprints/CN/SPRINT_PLAN_CN.md` (committed vs stretch, capacity, velocity)
- `.agile-v/sprints/CN/RETRO_CN.md` (retrospective, CR-XXXX generation)

**Why It Matters:** Bridges formal V-Model rigor with practical sprint-based delivery. Teams can deliver incrementally while preserving full traceability. No "backlog item without a REQ" — if it's not traced, it's either missing a requirement or technical debt.

**Example Sprint Flow:**
```
REQ-0001 to REQ-0010 (CRITICAL) → Backlog grooming → 
BL-0001a, BL-0001b, BL-0002, BL-0003 (Sprint 1, 25 points) → 
Sprint execution → Sprint review (REQ-0001, REQ-0002 verified) → 
Retro (issue found) → CR-0001 (Clarify REQ-0004) → Cycle 2
```

### 5. Release-Manager Skill (`release-manager/`)

**Purpose:** Extends rigor beyond Human Gate 2 into production deployment.

**Key Features:**
- Rollout plans (big bang, phased, canary, blue-green) with validation gates per phase
- Rollback procedures with measurable rollback triggers (error rate >1%, latency >1s, etc.)
- Pre-release checklist (Gate 2 complete, infrastructure ready, monitoring active, stakeholder approval)
- Post-release validation (verify every REQ in production)
- Incident response: production issues generate INC-XXXX → CAPA-XXXX → CR-XXXX for next cycle

**New Artifacts:**
- `.agile-v/RELEASE_PLAN_CN.md` (deployment strategy, rollout phases, risk assessment)
- `.agile-v/RELEASE_NOTES_vX.Y.Z.md` (user-facing changelog with REQ traceability)
- `.agile-v/ROLLBACK_PLAN.md` (rollback procedure, triggers, data recovery)

**Why It Matters:** Completes the pipeline. Stage 5 (Acceptance) now has formal "Release" and "Operate" phases. "Verified" becomes "Deployed" with same rigor as pre-production.

**Example Phased Rollout:**
```
Gate 2 approval → Release Plan (canary 1% → 10% → 50% → 100%) → 
Phase 1 (1%, 1 hour): metrics pass → Phase 2 (10%, 2 hours): metrics pass → 
Phase 3 (50%, 4 hours): metrics pass → Phase 4 (100%) → 
Post-release validation (all REQs verified in prod) → Release complete
```

### 6. Observability-Planner Skill (`observability-planner/`)

**Purpose:** Production monitoring as continuous requirements validation.

**Key Features:**
- Every metric (MET-XXXX) maps to a REQ-XXXX (performance, availability, correctness)
- Alerts trigger on REQ violation thresholds (e.g., REQ-0015: Dashboard ≤3s → Alert if p95 >3s)
- SLOs and error budgets (99.9% availability = 40 min downtime/month)
- Dashboards include "Requirement Validation" panel (is each REQ still satisfied in production?)
- Incidents feed back: INC-XXXX → root cause analysis → CR-XXXX → next cycle

**New Artifacts:**
- `.agile-v/OBSERVABILITY_PLAN.md` (MET-XXXX with REQ mappings, alert rules, SLOs)
- Dashboard definitions (System Health, Requirement Validation, Business Metrics)
- Runbooks (per-alert incident response procedures)

**Why It Matters:** Closes the feedback loop. Requirements aren't just verified pre-release — they're continuously monitored in production. Production anomalies drive requirements improvements, not just firefighting.

**Example:**
```
REQ-0015 (Dashboard load ≤3s) → MET-0001 (HTTP latency histogram) → 
Dashboard panel (p95 latency time series) → Alert ALR-0002 (p95 >3s for 5 min) → 
Alert fires → On-call executes runbook → Investigation finds N+1 query → 
INC-0001 logged → CR-0001 created (update REQ-0018: add query count limit) → Cycle 2
```

## Full Lifecycle Traceability Chain

Version 1.5 enables end-to-end traceability from initial user insight to production incident:

```
User Research (OBS-XXXX) → 
Insight (INS-XXXX) → 
Hypothesis (HYP-XXXX) → 
Experiment (EXP-XXXX) → 
Candidate REQ (CANDIDATE-XXXX) → 
Formal Requirement (REQ-XXXX) → 
Backlog Item (BL-XXXX) → 
Sprint Commitment (SPRINT_PLAN_CN.md) → 
Build Artifact (ART-XXXX) → 
Test Case (TC-XXXX) → 
Verification (VALIDATION_SUMMARY.md) → 
Human Gate 2 Approval (APPROVALS.md) → 
Release (RELEASE_PLAN_CN.md) → 
Production Metric (MET-XXXX) → 
Incident (INC-XXXX) → 
Change Request (CR-XXXX) → 
Next Cycle (CHANGE_LOG.md)
```

Every step is traceable, auditable, and versioned.

## Workflow Examples

### Workflow 1: New Project from Discovery

1. **discovery-analyst** — User interviews → OBS, INS, HYP → CANDIDATE-XXXX
2. **threat-modeler** — STRIDE analysis → THREAT-XXXX → CANDIDATE-SEC-XXXX
3. **ux-spec-author** — Design flows → A11Y, PERF → CANDIDATE-XXXX
4. **Human Gate 0** — Approve discovery artifacts
5. **requirement-architect** — Formalize all CANDIDATE-XXXX → REQ-XXXX
6. **logic-gatekeeper** — Validate requirements
7. **Human Gate 1** — Approve blueprint
8. **build-agent-\*** + **test-designer** — Parallel synthesis
9. **red-team-verifier** — Execute tests
10. **Human Gate 2** — Approve validation
11. **release-manager** — Plan rollout
12. **observability-planner** — Define metrics/alerts
13. **Deploy to production** — Monitor via observability

### Workflow 2: Sprint-Based Delivery

1. **requirement-architect** + **logic-gatekeeper** — Full REQ set (Gate 1 approved)
2. **agile-v-product-owner** — Decompose into BACKLOG.md
3. For each sprint (e.g., 3 sprints in Cycle 1):
   - **Sprint Planning** — Commit to BL-XXXX (maps to REQ-XXXX)
   - **build-agent-\*** — Build sprint scope
   - **test-designer** — Design tests for sprint scope
   - **red-team-verifier** — Verify sprint artifacts
   - **Sprint Review** — REQs verified this sprint
   - **Retrospective** — Generate CR-XXXX if needed
4. After all sprints → **Human Gate 2** → **Release**

### Workflow 3: Production Incident Response

1. **Incident detected** — observability-planner alert fires (MET-XXXX violates threshold)
2. **Triage** — On-call executes runbook
3. **Mitigation** — release-manager executes rollback plan
4. **Resolution** — Verify metrics return to baseline
5. **Post-Incident** — observability-planner logs INC-XXXX
6. **Root Cause Analysis** — Identify REQ gap
7. **Change Request** — agile-v-lifecycle creates CR-XXXX
8. **Cycle N+1** — requirement-architect updates REQ-XXXX → re-enter pipeline

## Compatibility with Existing Skills

All new skills integrate seamlessly with existing Agile V v1.0-1.4 skills:

- **`discovery-analyst`** → feeds → **`requirement-architect`**
- **`threat-modeler`** → feeds → **`requirement-architect`** (CRITICAL priority)
- **`ux-spec-author`** → feeds → **`requirement-architect`** (design constraints)
- **`agile-v-product-owner`** → orchestrates → **`build-agent-*`**, **`test-designer`**, **`red-team-verifier`** (per sprint)
- **`release-manager`** → uses → **`red-team-verifier`** output (VALIDATION_SUMMARY.md), **`observability-planner`** dashboards
- **`observability-planner`** → monitors → **REQ-XXXX** (from **`requirement-architect`**), triggers → **CR-XXXX** (to **`agile-v-lifecycle`**)

No breaking changes. Existing projects can adopt new skills incrementally.

## Migration Guide

### For Existing Agile V v1.0-1.4 Projects

**No migration required.** New skills are additive.

**Optional Enhancements:**

1. **Add discovery phase (before Cycle 1):**
   - Run `discovery-analyst` on existing user research/feedback
   - Backfill `DISCOVERY_LOG.md` with lineage for existing REQs
   - Run `threat-modeler` to identify missing security REQs

2. **Adopt sprint-based delivery (within current cycle):**
   - Use `agile-v-product-owner` to decompose remaining REQs into sprints
   - Continue with existing pipeline per sprint

3. **Enhance release process (before next release):**
   - Use `release-manager` to formalize rollout plan
   - Use `observability-planner` to define production metrics

4. **Strengthen security (before next cycle):**
   - Run `threat-modeler` to identify threats
   - Add security REQs as CRITICAL priority

### For New Projects

**Recommended workflow:**
1. Start with `discovery-analyst`, `threat-modeler`, `ux-spec-author` (discovery phase)
2. `requirement-architect` formalizes all candidate REQs
3. `agile-v-product-owner` decomposes into sprints (or run full cycle at once)
4. Standard pipeline: build → test → verify → Gate 2
5. `release-manager` + `observability-planner` for production deployment
6. Monitor, iterate, repeat

## Breaking Changes

## Deprecations

**None.**

## Known Limitations

1. **Discovery phase is optional** — Projects with clear requirements can skip `discovery-analyst`.
2. **Sprint-based delivery is optional** — Projects can continue with single-cycle execution.
3. **Human Gate 0** is a new checkpoint — teams must decide when discovery is "approved."
4. **Production deployment skills** (`release-manager`, `observability-planner`) assume cloud/DevOps infrastructure; adapt for on-premise or manual deployments.
